# Анализ безопасности процесса авторизации и регистрации
**Проект:** chat-service
**Аналитик:** Antigravity (Кибербезопасность)

## 1. Общая оценка (Executive Summary)
В ходе аудита исходного кода модулей `app/routers/auth.py` и `app/security.py` был выявлен ряд архитектурных и логических уязвимостей в механизмах аутентификации, регистрации и управления сессиями. Текущая реализация не соответствует современным стандартам безопасности (OWASP) и подвержена ряду атак, таких как перебор паролей, атаки по времени (timing attacks) и компрометация сессий.

Процесс авторизации действительно нуждается в серьезной переработке. Ниже представлен детальный разбор найденных проблем.

---

## 2. Уязвимости процесса авторизации (Login)

### 2.1. Уязвимость к атакам по времени (Timing Attack) для перечисления пользователей
**Файл:** `app/routers/auth.py`, функция `login`
**Описание:** 
В текущем коде проверка пароля происходит следующим образом:
```python
if not user or not verify_password(form.password, user.password_hash):
```
Алгоритм `bcrypt` специально спроектирован так, чтобы работать медленно. Если пользователя `form.username` нет в базе, переменная `user` равна `None`, и функция `verify_password` **не вызывается** (благодаря short-circuit вычислению логического "или"). Это означает, что ответ на запрос с несуществующим логином вернется в разы быстрее, чем ответ на запрос с существующим логином, но неверным паролем.
**Риск:** Злоумышленник может измерять время ответа сервера и безошибочно составить список существующих в системе логинов.

### 2.2. Отсутствие защиты от подбора паролей (Brute-force & Credential Stuffing)
**Описание:** Эндпоинты `/login` и `/register` никак не ограничивают количество запросов от одного IP-адреса или для одного логина.
**Риск:** Злоумышленники могут использовать автоматизированные инструменты для подбора паролей (brute-force) или перебора слитых баз (credential stuffing) без каких-либо препятствий. Отсутствует механизм блокировки аккаунта (Account Lockout) или троттлинга (Rate Limiting).

## 3. Проблемы с криптографией и хешированием паролей

### 3.1. Небезопасное обрезание паролей (Bcrypt Truncation)
**Файл:** `app/security.py`, функции `hash_password` и `verify_password`
**Описание:** 
```python
secret = plain.encode()[:72]
```
Алгоритм bcrypt действительно имеет ограничение на длину пароля в 72 байта. Однако, "тихое" обрезание пароля — это плохая практика. Если пользователь введет пароль длиной 100 символов, система сохранит и будет проверять только первые 72. Таким образом, любой пароль, у которого совпадают первые 72 символа с оригинальным, будет считаться верным.
**Решение:** Вместо обрезания рекомендуется предварительно хешировать пароль алгоритмом SHA-256 (или SHA-512), а затем результат передавать в bcrypt: `bcrypt.hashpw(hashlib.sha256(password.encode()).hexdigest().encode(), salt)`.

## 4. Уязвимости процесса регистрации

### 4.1. Слабая парольная политика
**Файл:** `app/routers/auth.py`, функция `register`
**Описание:** Пароль проверяется только на минимальную длину (8 символов). Отсутствуют проверки на максимальную длину (предотвращение DoS атак при генерации хеша), сложность (наличие цифр, спецсимволов, регистров) и проверку по базам утекших паролей.

### 4.2. Недостаточная валидация имени пользователя
**Описание:** Имя пользователя проверяется только на длину (от 3 до 32 символов). Не фильтруются спецсимволы, пробелы и невидимые символы (Unicode homoglyphs), что может привести к атакам типа "имперсонация" (когда злоумышленник создает никнейм, визуально неотличимый от никнейма администратора) или проблемам с инъекциями в других частях приложения.

---

## 5. Уязвимости управления сессиями (JWT)

### 5.1. Чрезмерно долгое время жизни Access-токена
**Файл:** `app/security.py`
**Описание:** `JWT_EXPIRE_MINUTES = 60 * 24 # 24 часа`.
Время жизни access-токена в 24 часа является слишком долгим. Если злоумышленник перехватит токен (например, через XSS-уязвимость на клиенте), он будет иметь полный доступ к аккаунту целые сутки.

### 5.2. Отсутствие механизма Refresh-токенов и отзыва сессий
**Описание:** В системе реализованы только долгоживущие Access-токены. При таком подходе невозможно принудительно завершить сессию пользователя (например, при смене пароля или нажатии кнопки "Выйти" со всех устройств). Кнопка "Logout" на клиенте просто удалит токен из памяти браузера, но сам токен останется валидным до истечения своего срока на стороне сервера.

### 5.3. Небезопасное хранение секретных ключей
**Файл:** `app/security.py`, `_load_or_create_key`
**Описание:** Ключи шифрования (`jwt_secret.key` и `msg_fernet.key`) генерируются и хранятся в файлах на диске рядом с кодом приложения. При уязвимости LFI (Local File Inclusion) или взломе сервера ключи будут мгновенно скомпрометированы, что позволит злоумышленнику выпускать поддельные JWT токены от лица любого пользователя (включая администраторов) и читать зашифрованную переписку. Ключи должны передаваться исключительно через переменные окружения (Environment Variables) или секретные хранилища (Vault).

### 5.4. Недостаточные Claims в JWT
**Описание:** В полезной нагрузке (payload) JWT токена присутствуют только `sub` и `exp`. Отсутствуют такие важные поля как `iat` (время выдачи токена), `jti` (уникальный идентификатор токена для реализации Blacklist) и `iss`/`aud` (верификация издателя и аудитории токена).

## 6. Рекомендации по исправлению

1. **Защита от Timing Attacks:** В функции `login`, если пользователь не найден в базе данных, необходимо вычислять фиктивный хеш `bcrypt` (dummy hash), чтобы время выполнения функции всегда было примерно одинаковым:
   ```python
   user = result.scalars().first()
   if not user:
       # Dummy проверка для выравнивания времени ответа
       verify_password(form.password, DUMMY_HASH)
       raise HTTPException(...)
   elif not verify_password(form.password, user.password_hash):
       raise HTTPException(...)
   ```
2. **Внедрение Rate Limiting:** Установить ограничение на количество попыток авторизации и регистрации (например, используя `slowapi` или кастомный middleware с Redis).
3. **Безопасное хеширование пароля:** Перейти на схему `bcrypt(sha256(password))` для снятия лимита в 72 байта и предотвращения коллизий усеченных строк.
4. **Ужесточение политики ввода:** Ввести строгую регулярку для имени пользователя (например, `^[a-zA-Z0-9_-]+$`) и усложнить требования к паролю.
5. **Модернизация управления сессиями:**
   - Сократить время жизни Access-токена до 10-15 минут.
   - Внедрить механизм Refresh-токенов с их хранением в базе данных (с возможностью их отзыва).
   - Вынести все криптографические ключи (`JWT_SECRET_KEY`) в переменные окружения (`.env` файл, который не попадает в систему контроля версий).
6. **Безопасная передача:** Убедиться, что сервис работает строго по HTTPS, иначе все токены могут быть перехвачены атакой Man-in-the-Middle.
